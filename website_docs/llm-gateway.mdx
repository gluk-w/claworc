## Overview

Claworc includes a built-in **LLM gateway** — an internal proxy that sits between your agent instances and LLM provider APIs (Anthropic, OpenAI, Ollama, and more).

Instead of giving each container a real API key, the control plane issues every instance a **virtual key** — a unique, per-instance, per-provider credential. The instance uses this virtual key to call the gateway. The gateway validates the key, looks up the real provider credential, and forwards the request.

**Your real API keys never leave the control plane.**

### Security properties

- **Credentials stay out of containers.** Even if an agent container is compromised, there are no real API keys inside it to steal. A container breakout or log leak cannot expose your provider credentials.
- **Encrypted at rest.** All provider API keys — both global settings and per-instance overrides — are stored Fernet-encrypted in the database. The encryption key is auto-generated on first run and stored separately.
- **Network isolation via SSH tunnel.** The gateway listens on `127.0.0.1` only and is never exposed on a public port. Containers reach it through a reverse SSH tunnel established by the control plane. There is no path to the gateway from outside the host.
- **Per-instance revocation.** Virtual keys are one-to-one: one key per instance per provider. You can remove a provider from one instance, or delete a provider entirely, without affecting any other instance or rotating your real provider key.
- **Audit trail.** Every request through the gateway is logged with timing, token counts, status codes, and error details.

## Configuring providers

Go to **Settings** and find the **Model API Keys** section.

### Adding a provider

Click **Add Provider**. A modal opens with two paths:

**From the catalog** — Select a known provider (Anthropic, OpenAI, Together AI, and others) from the dropdown. Claworc pre-fills the name and fetches the model list automatically from the Claworc hosted catalog. You only need to supply your API key for that provider.

**Custom** — Select **Custom** from the dropdown to configure any provider manually, including self-hosted models (Ollama, vLLM, LM Studio) or any service with an OpenAI-compatible API. Fill in:

| Field | Description |
|-------|-------------|
| **Name** | Display name shown in the dashboard |
| **Base URL** | Provider API base URL — see [API types](#api-types) for the correct format per provider |
| **API type** | Request format the provider uses — controls auth headers and URL handling |
| **API key** | Provider credential. Stored encrypted and shown masked (`****` + last 4 characters) after saving. |
| **Models** | For custom providers: define each model with its ID, name, and optional metadata (context window, max tokens, cost rates) |

The provider **key** — a URL-safe slug like `anthropic` or `my-ollama` — is derived automatically from the name when you create the provider. It cannot be changed after creation.

### Per-instance API key overrides

The API key you enter in Settings is the **global** key for that provider — it applies to all instances that have the provider enabled. If a specific instance needs to use a different key (for example, a separate billing account), you can set a per-instance override on that instance's detail page.

The lookup order is: **instance override first, then global setting.** The global key is the fallback whenever no instance-level override exists.

### Editing a provider

Click the pencil icon next to any provider row. You can update the name, base URL, API type, model list, or rotate the API key. The provider key is immutable.

### Deleting a provider

Open the edit modal and click **Delete**. Deleting a provider immediately revokes all virtual keys issued to instances for that provider. Any instance that had this provider enabled will lose access to it.

### Syncing catalog models

For catalog-backed providers, the model list is fetched from the Claworc hosted catalog and cached for one hour. Click **Sync Models** to force a refresh for all catalog providers at once. You can also trigger a sync for a single provider via the API (`POST /api/v1/llm/providers/{id}/sync`).

## API types

The `api_type` field controls which authentication header the gateway forwards and how it handles the base URL. Choose the type that matches your provider's API format.

| API type | Typical providers | Auth sent upstream | Base URL format |
|----------|------------------|-------------------|-----------------|
| `openai-completions` | OpenAI, Together AI, OpenRouter, and any OpenAI-compatible API | `Authorization: Bearer` | Must end with `/v1` — e.g. `https://api.openai.com/v1` |
| `openai-responses` | OpenAI Responses API | `Authorization: Bearer` | Must end with `/v1` |
| `anthropic-messages` | Anthropic | `x-api-key` | Root URL only — e.g. `https://api.anthropic.com` (no `/v1`) |
| `google-generative-ai` | Google Gemini | `x-goog-api-key` | Include `/v1beta` — e.g. `https://generativelanguage.googleapis.com/v1beta` |
| `ollama` | Ollama (self-hosted) | `Authorization: Bearer` (empty key accepted) | Root URL — e.g. `http://localhost:11434` |

### Base URL notes

- **`anthropic-messages`**: set the root URL without `/v1` (e.g. `https://api.anthropic.com`). The Anthropic SDK appends `/v1/messages` itself.
- **`google-generative-ai`**: include `/v1beta` in the URL. The Google SDK omits the version segment when a custom base URL is provided, so you must include it explicitly.
- **`openai-completions` / `openai-responses`**: the URL must end with `/v1`. The gateway de-duplicates the segment if it is already present.
- **`ollama`**: an empty or missing API key is accepted — Ollama does not require authentication by default.

Any provider with an OpenAI-compatible `/chat/completions` endpoint can be added as a custom provider with `api_type: openai-completions`, regardless of whether it appears in the catalog.

## Enabling providers on an instance

Open an instance and go to the **Overview** tab. The **Enabled Models** section (visible to admins) shows which providers and models are active for that instance.

Click **Edit** to enter edit mode. For each provider you want to enable, click its row to expand it:

- **Catalog providers and custom providers with a model list** — an accordion shows all available models. Check individual models, or use **Select all** / **Deselect all**.
- **Dynamic providers** (no fixed model list, e.g. Ollama with a local model server) — a toggle enables or disables the provider. Models are determined at runtime.

After selecting models, pick a **Default model** from the dropdown. This becomes the primary model OpenClaw uses for new agents unless the agent explicitly requests a different one.

Click **Save**. Claworc pushes the configuration to the container over SSH in the background. A toast notification confirms when the update completes or reports any error.

<Note>
  The instance must be running and SSH-connected for the configuration to be pushed immediately. If the instance is stopped when you save, the configuration will be applied the next time it starts and connects.
</Note>

## How it works end-to-end

When you enable a provider for an instance, Claworc:

1. **Issues a virtual key.** A unique `claworc-vk-<48 hex chars>` token is generated for that instance and provider pair. The virtual key is stored in the database and never shown in the UI.
2. **Opens a tunnel.** The control plane creates a reverse SSH tunnel into the container. The container can now reach the LLM gateway at `http://127.0.0.1:<port>` — but only from inside the container over that tunnel.
3. **Configures OpenClaw.** The control plane writes provider configuration into the container via SSH, pointing each enabled provider at the local gateway address with the virtual key as the credential. The real provider key is never sent to the container.

When the instance makes an LLM request, the flow is:

1. OpenClaw sends the request to `http://127.0.0.1:<port>` using the virtual key.
2. The gateway validates the virtual key and looks up which provider and instance it belongs to.
3. The gateway resolves the real API key (instance override if present, otherwise global setting).
4. The gateway strips incoming auth headers, sets the correct outgoing auth header for the provider's `api_type`, and forwards the request.
5. The response is proxied back to OpenClaw, and the request is recorded in the usage log.

## Usage logs

Every request proxied through the gateway is recorded. Query them via the API:

```
GET /api/v1/llm/usage
```

| Parameter | Type | Description |
|-----------|------|-------------|
| `instance_id` | integer | Filter to one instance |
| `provider_id` | integer | Filter to one provider |
| `limit` | integer | Max results (default 100, max 1000) |
| `offset` | integer | Pagination offset |

Results are ordered newest-first.

| Field | Description |
|-------|-------------|
| `model_id` | Model ID from the request |
| `provider_key` | Provider slug (e.g. `anthropic`) |
| `input_tokens` | Prompt tokens consumed |
| `output_tokens` | Completion tokens generated |
| `cached_input_tokens` | Tokens served from the provider's prompt cache |
| `cost_usd` | Estimated cost in USD based on configured model rates |
| `status_code` | HTTP status returned by the upstream provider |
| `latency_ms` | End-to-end latency in milliseconds |
| `error_message` | First 500 bytes of error body on 4xx/5xx responses |
| `requested_at` | UTC timestamp |

<Note>
  Streaming responses (`text/event-stream`) are logged with zero token counts and zero cost — the response body is forwarded directly without buffering, so token counts are not available.
</Note>

Token counts and cost are only populated for non-streaming responses where the provider returns usage data in the response body. The `cost_usd` field is zero when the model does not have cost rates configured.

**Cost formula (non-streaming):**

```
cost_usd = (
    (input_tokens − cached_input_tokens) × input_rate  +
    cached_input_tokens                  × cache_read_rate  +
    output_tokens                        × output_rate
) / 1,000,000
```

Rates come from the `cost` field of the matching model in the provider's model list. Configure rates when adding or editing a custom provider, or when editing catalog provider models.

<Info>
  Per-instance and per-virtual-key usage breakdowns are planned for a future release.
</Info>

Usage logs are stored in a dedicated file (`llm-logs.db`) in the data directory, separate from the main `claworc.db`. This keeps the primary database lean and allows log rotation independently of other data.

## API reference

All provider and usage endpoints require admin authentication.

### Providers

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/llm/providers` | List all configured providers |
| `POST` | `/api/v1/llm/providers` | Create a provider |
| `PUT` | `/api/v1/llm/providers/{id}` | Update a provider |
| `DELETE` | `/api/v1/llm/providers/{id}` | Delete a provider and revoke its virtual keys |
| `POST` | `/api/v1/llm/providers/{id}/sync` | Sync model list from catalog for one provider |
| `POST` | `/api/v1/llm/providers/sync` | Sync model lists for all catalog providers |

### Catalog

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/llm/catalog` | List all providers in the Claworc model catalog |
| `GET` | `/api/v1/llm/catalog/{key}` | Get models and metadata for a specific catalog provider |

### Usage logs

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/llm/usage` | Query usage logs with optional filters |

### Provider fields

When creating (`POST`) or updating (`PUT`) a provider:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `key` | string | Yes (create only) | URL-safe slug, e.g. `anthropic` or `my-ollama`. Derived from the name automatically in the UI. Immutable after creation. |
| `name` | string | Yes | Display name |
| `base_url` | string | Yes | Provider base URL — format depends on `api_type` |
| `api_type` | string | No | Default: `openai-completions` |
| `provider` | string | No | Catalog provider key (e.g. `anthropic`) if backed by the catalog; empty for custom |
| `models` | array | No | Model definitions for custom providers |

Each entry in `models`:

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Model ID used in API calls (e.g. `claude-sonnet-4-6`) |
| `name` | string | Display name |
| `reasoning` | boolean | Whether the model supports extended reasoning |
| `contextWindow` | integer | Context window in tokens |
| `maxTokens` | integer | Maximum output tokens |
| `cost` | object | Per-million-token rates: `{ input, output, cacheRead, cacheWrite }` |
